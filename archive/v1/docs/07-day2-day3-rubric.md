# 🏁 เกณฑ์ให้คะแนน Prototype & Pitch Assessment (70 คะแนน)

> Day 2–3 = 70 คะแนน / 100 ทั้งโครงการ
> ใช้ rubric นี้ดูว่าทีมทำ prototype ที่พิสูจน์ไอเดียได้จริงไหม, มีหลักฐานครบไหม, และเล่าให้คนอื่นเข้าใจได้ไหม

Day 2 เน้น: เลือก scope ให้เล็กพอ, build ให้ทำงานจริง, เก็บ test/evidence ให้ครบ

Day 3 เน้น: demo ให้ชัด, pitch ให้เข้าใจง่าย, ตอบ Q&A จากหลักฐานจริง

---

## 📋 Rubric Overview

| หัวข้อ | น้ำหนัก | คะแนนเต็ม |
|---|---|---|
| 1. MVP Scope & Problem Fit | 2.5 | 10 |
| 2. Working Prototype & Integration | 5 | 20 |
| 3. Testing, Evidence & Iteration | 3.75 | 15 |
| 4. Demo Clarity & User Experience | 2.5 | 10 |
| 5. Pitch & Q&A | 2.5 | 10 |
| 6. Resource & Token Management | 1.25 | 5 |

ระดับการประเมิน: **4 (ดีมาก) | 3.0-3.9 (ดี) | 2.0-2.9 (พอใช้) | 1.0-1.9 (ต้องปรับปรุง)**

## 1. MVP Scope & Problem Fit (10 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | เลือกปัญหา ผู้ใช้ และฟังก์ชันหลักได้ชัดเจน อธิบายได้ว่าทำไม Edge AI เหมาะ และ scope แบบ Bronze / Silver / Gold สมเหตุสมผลกับเวลาที่มี | Updated Idea Canvas, Prototype Scope Sheet, System Architecture, Success Metrics |
| **3** | เป้าหมายค่อนข้างชัด มีฟังก์ชันหลักชัดเจนแล้ว แต่ยังมีบางส่วนที่กว้างเกินไปหรือยังตอบเรื่องผู้ใช้ไม่สุด | Idea Canvas ครบแต่ยังมี feature เกินจำเป็น |
| **2** | รู้ว่าจะทำอะไรคร่าวๆ แต่ยังแยก core function ออกจาก feature เสริมไม่ได้ | อธิบายปัญหาได้ แต่ยังตอบไม่ได้ว่าจะสาธิตอะไรเป็นหลัก |
| **1** | ไอเดียยังไม่ชัด เปลี่ยนเป้าหมายไปมา หรือ scope ใหญ่เกินเวลาจนทำให้ prototype เดินต่อไม่ได้ | ไม่มี scope ที่ชัด หรือยังไม่มีภาพระบบรวม |

**Tips สำหรับนักเรียน:**
- ถ้าตอบคำถามว่า "ฟังก์ชันเดียวที่ต้องสาธิตให้ได้คืออะไร" ไม่ได้ใน 1 ประโยค แปลว่า scope ยังไม่คมพอ
- ปัญหาที่ชัด + demo ที่จบจริง ได้คะแนนง่ายกว่าระบบใหญ่ที่ยังไม่รวมกัน

## 2. Working Prototype & Integration (20 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | Prototype ทำงานจริงครบเส้นทางตั้งแต่ input → AI → UNO Q → output ฟังก์ชันหลักเสถียรพอสำหรับ demo และมีแผนสำรองเมื่อบางส่วนพัง | Live demo, source code, photo/video ตอนรันจริง, output ตอบสนองตามโจทย์ |
| **3** | ระบบหลักทำงานได้เกือบครบ มีสะดุดเล็กน้อย แต่ยังสาธิตฟังก์ชันหลักได้ | Prototype ทำงานได้ แต่ยังไม่นิ่งหรือยังต้อง reset บางครั้ง |
| **2** | บางส่วนทำงานได้ เช่น model ทำนายได้หรือ output ติด แต่ยังรวมระบบไม่ครบ | มี inference หรือ output แยกกัน แต่ยังไม่ต่อเป็น flow เดียว |
| **1** | ระบบไม่สามารถสาธิตฟังก์ชันหลักได้ หรือยังอยู่ในระดับชิ้นส่วนแยกที่ไม่เชื่อมกัน | Train ได้แต่ deploy ไม่ได้, deploy ได้แต่ output ไม่ทำงาน |

**Tips สำหรับนักเรียน:**
- ปิด Bronze ให้ได้ก่อน แล้วค่อยเพิ่ม Silver / Gold
- ถ้าระบบยังไม่วิ่งจริง อย่าเพิ่งเสียเวลาไปกับของตกแต่งหรือ dashboard

## 3. Testing, Evidence & Iteration (15 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | ทดสอบอย่างน้อย 5 ครั้งในหลายเงื่อนไข บันทึกผลเป็นระบบ มี Debug Log และใช้ผลทดสอบไปปรับ prototype หรือ model รอบถัดไป | Test Log V2, Debug Log, screenshot prediction/confidence, V1 vs V2 notes, demo video |
| **3** | มีการทดสอบและบันทึกผลชัดเจน แต่ยังมีกรณีทดสอบไม่หลากหลายหรือ iterate ยังไม่มากพอ | Test Log ครบขั้นต่ำ แต่ยังไม่มี edge case ชัด |
| **2** | มีการทดสอบเบื้องต้น แต่หลักฐานยังไม่พอจะพิสูจน์ว่า prototype ใช้ได้จริง | มี test บางครั้งแต่จดไม่ครบ, ไม่มี debug reasoning |
| **1** | ไม่มีหลักฐานการทดสอบจริง หรืออ้างผลโดยไม่มีบันทึก | ไม่มี Test Log หรือไม่มีหลักฐานว่าเคยรันจริง |

**Tips สำหรับนักเรียน:**
- เก็บทั้ง easy case และ case ที่ระบบพลาด ไม่ใช่เก็บแต่ตอนระบบทายถูก
- หลักฐานที่ดีช่วยทั้งคะแนน debug, demo, และ Q&A พร้อมกัน

## 4. Demo Clarity & User Experience (10 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | Demo เข้าใจง่ายภายใน 1-2 นาที เห็น input → output ชัด ผู้ชมดูแล้วเข้าใจทันทีว่าระบบช่วยใครและช่วยอย่างไร | Demo script, scenario table, video 15-30 วินาที, form factor หรือ mock setup |
| **3** | Demo ทำงานได้และเข้าใจได้เป็นส่วนใหญ่ แต่ลำดับยังไม่คม หรือผู้ชมต้องถามเพิ่มเพื่อเข้าใจ | Demo ผ่านแต่ยังเล่า flow ไม่ลื่น |
| **2** | Demo ยังไม่ชัด เห็นระบบทำงานบางส่วน แต่ยังไม่เห็นคุณค่าจากมุมผู้ใช้ | โชว์ output ได้แต่ไม่เห็นสถานการณ์ใช้งาน |
| **1** | Demo สับสน หรือไม่สามารถโชว์ฟังก์ชันหลักได้จริง | ผู้ชมไม่เข้าใจว่าเกิดอะไรขึ้นหรือระบบใช้ทำอะไร |

**Tips สำหรับนักเรียน:**
- Demo ที่ดีไม่ใช่พูดยาว แต่คือทำให้คนดูเห็นเหตุการณ์ก่อน แล้วค่อยเฉลยว่าระบบทำอะไร
- ถ้าโชว์สดเสี่ยง ให้เตรียม video สั้นเป็นแผนสำรองเสมอ

## 5. Pitch & Q&A (10 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | เล่า Problem, User, Solution, Technology, Result, Limitation และ Next Step ได้ครบ ตอบ Q&A ด้วยเหตุผลและอ้างอิงหลักฐานจริงได้ | Pitch draft, speaking roles, evidence folder, คำตอบเรื่องข้อจำกัดและการตัดสินใจ |
| **3** | Pitch ชัดเจนในภาพรวม แต่บางจุดยังอธิบายไม่ลึกพอ หรือ Q&A ยังตอบได้ไม่ครบทุกมุม | พูด flow ได้ แต่ยังไม่ชัดเรื่อง limitation หรือ trade-off |
| **2** | เล่าไอเดียได้ แต่ยังโยงกับ prototype หรือหลักฐานไม่แน่นพอ | พูดได้ว่าทำอะไร แต่ยังตอบไม่ได้ว่าทำไมเลือกวิธีนี้ |
| **1** | Pitch กระจัดกระจาย หรือ Q&A ตอบจากการเดาโดยไม่มีหลักฐานรองรับ | Demo กับ pitch ไม่ตรงกัน, ตอบไม่ได้ว่า AI ผิดเมื่อไหร่ |

**Tips สำหรับนักเรียน:**
- เวลาตอบ Q&A ให้เริ่มจากสิ่งที่ทดลองจริงก่อน แล้วค่อยพูดสิ่งที่อยากทำต่อ
- คำว่า "ยังไม่ได้ลอง" ดีกว่าตอบเดา ถ้าตามด้วยสิ่งที่จะทดลองต่ออย่างชัดเจน

## 6. Resource & Token Management (5 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | ใช้ token และอุปกรณ์อย่างตั้งใจ ทุกชิ้นมีเหตุผลรองรับ เชื่อมกับการพิสูจน์ไอเดีย และทีมแบ่งบทบาททำงานได้สมดุล | Token Usage Log, รายการอุปกรณ์ที่ใช้จริง, commit จากหลายคน, role assignment |
| **3** | ใช้ทรัพยากรค่อนข้างเหมาะสม มีเหตุผลรองรับเป็นส่วนใหญ่ แต่ยังมีของบางชิ้นที่ใช้ไม่คุ้ม | ใช้อุปกรณ์เกือบครบตามแผน แต่ยังอธิบายเหตุผลบางชิ้นไม่ชัด |
| **2** | มีการใช้ทรัพยากรเกินจำเป็น หรือวางแผนการใช้งานยังไม่ชัด | เบิกของไว้ก่อนแต่ไม่ได้ใช้จริง, token log ไม่ครบ |
| **1** | ใช้ทรัพยากรแบบไม่มีแผน ทำให้เสียเวลา เสีย token หรือกระทบงานหลัก | เบิกของจำนวนมากแต่ไม่ช่วยให้ demo ดีขึ้น |

**Tips สำหรับนักเรียน:**
- Token ที่ใช้คุ้ม = อุปกรณ์ที่ทำให้ prototype พิสูจน์ไอเดียได้ชัดขึ้นจริง
- ถ้าของชิ้นหนึ่งตัดออกแล้ว demo ยังเหมือนเดิม แปลว่าอาจไม่จำเป็น


## ⚠️ Red Flags (ลดคะแนน)

- ❌ ใส่ feature เยอะ แต่ฟังก์ชันหลักยังไม่ทำงานจริง
- ❌ Demo กับ pitch พูดคนละเรื่อง
- ❌ อ้างว่าแม่น แต่ไม่มี test log หรือ evidence รองรับ
- ❌ ใช้อุปกรณ์เพิ่มเพราะดูเท่ ไม่ได้ช่วยพิสูจน์ไอเดีย
- ❌ ตอบ Q&A แบบเดา ทั้งที่ยังไม่เคยทดลองจริง
