# 📊 เกณฑ์ให้คะแนน Skill Assessment (30 คะแนน)

> Day 1 = 30 คะแนน / 100 ทั้งโครงการ
> ใช้ rubric นี้เช็กตัวเองว่าถ้าทีมอยากได้คะแนนจาก "กระบวนการ" ต้องมีหลักฐานอะไรและทำพฤติกรรมแบบไหนระหว่างวัน

---

## 📋 Rubric Overview

| หัวข้อ | น้ำหนัก | คะแนนเต็ม |
|---|---|---|
| 1. Setup & Safety | 1.25 | 5 |
| 2. Core Implementation | 2.5 | 10 |
| 3. Data & Testing | 1.25 | 5 |
| 4. Debug & Explain | 1.25 | 5 |
| 5. Participation & Collaboration | 1.25 | 5 |

ระดับการประเมิน: **4 (ดีมาก) | 3.0-3.9 (ดี) | 2.0-2.9 (พอใช้) | 1.0-1.9 (ต้องปรับปรุง)**

---

## 1. Setup & Safety (5 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | จัดเตรียมอุปกรณ์และเชื่อมต่อระบบได้ถูกต้อง ปลอดภัย เป็นระเบียบ และพร้อมใช้งาน | Commit setup ในชั่วโมงแรก, photo workspace |
| **3** | จัดเตรียมอุปกรณ์ได้ถูกต้องเป็นส่วนใหญ่ มีข้อผิดพลาดเล็กน้อยที่ไม่ส่งผลต่อการใช้งานหลัก | Setup เสร็จแต่ช้ากว่ากำหนด |
| **2** | จัดเตรียมอุปกรณ์ได้บางส่วน แต่ยังต้องได้รับคำแนะนำเพิ่มเติม | ต้องให้ TA ช่วย setup |
| **1** | จัดเตรียมอุปกรณ์ไม่ถูกต้อง หรือมีความเสี่ยงด้านความปลอดภัย | เสียบ Qwiic ผิด, ไม่จัดสาย |

**Tips สำหรับนักเรียน:**
- ถ่ายภาพ workspace แล้ว push เป็น `assets/workspace.jpg`
- เขียน setup ใน `docs/setup-notes.md`

## 2. Core Implementation (10 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | พัฒนาระบบได้ครบตามโจทย์ ฟังก์ชันหลักทำงานได้ถูกต้องและต่อเนื่อง | Model V1+V2, deploy ได้, ทดสอบบนบอร์ดจริง |
| **3** | ระบบทำงานได้เกือบครบตามโจทย์ มีข้อผิดพลาดเล็กน้อยที่ไม่กระทบการทำงานหลัก | V2 มี accuracy ดีกว่า V1 |
| **2** | ระบบสามารถทำงานได้บางฟังก์ชัน แต่ยังไม่ครบถ้วนตามโจทย์ | มี V1 deploy ได้ แต่ไม่ได้ iterate |
| **1** | ระบบไม่สามารถทำงานได้ หรือไม่สอดคล้องกับโจทย์หลัก | Train ไม่จบ / Deploy ไม่ได้ |

**Tips สำหรับนักเรียน:**
- Tag release `v1.0` และ `v2.0` ใน GitHub
- เขียนผลลัพธ์ใน README

---

## 3. Data & Testing (5 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | ทดสอบอย่างน้อย 2 สถานการณ์ บันทึกผลและนำผลการทดสอบมาปรับปรุงได้อย่างเหมาะสม | `W3-prediction-log.md` ≥10 cases + `docs/v1-vs-v2.md` |
| **3** | มีการทดสอบและสรุปผลได้ชัดเจน แต่ยังไม่ครอบคลุมทุกกรณี | มี Prediction Log แต่ไม่ได้ iterate |
| **2** | มีการทดสอบในระดับพื้นฐาน แต่หลักฐานหรือผลการทดสอบยังไม่ชัดเจน | มี Prediction Log <10 cases |
| **1** | ไม่มีการทดสอบหรือไม่สามารถอธิบายผลการทดสอบได้ | ไม่มี Prediction Log |

**Tips สำหรับนักเรียน:**
- **Prediction Log** สำคัญที่สุดในหัวข้อนี้
- ทดสอบ edge cases (เช่น แสงต่ำ, มุมแปลก) ไม่ใช่แค่ best case

---

## 4. Debug & Explain (5 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | สามารถวิเคราะห์และแก้ปัญหาได้อย่างเป็นขั้นตอน พร้อมอธิบายโค้ด เหตุผล และข้อจำกัดของระบบได้อย่างชัดเจน | Commit messages อธิบาย "ทำไม" + `docs/issues-and-fixes.md` |
| **3** | สามารถอธิบายแนวทางการแก้ปัญหาและการทำงานของระบบได้ค่อนข้างชัดเจน | มี issue tracking |
| **2** | สามารถแก้ปัญหาได้บางส่วน แต่อธิบายแนวคิดหรือการทำงานได้อย่างจำกัด | Commit messages ขาดรายละเอียด |
| **1** | ไม่สามารถแก้ปัญหาหรืออธิบายการทำงานได้ | ไม่มี commit หรือ commit ไม่สื่อ |

**Tips สำหรับนักเรียน:**
- Commit message **ห้ามใช้** "update", "fix", "test"
- ใช้ format: `<verb>: <what> เพราะ <why>`
  - ✅ `feat: เพิ่ม class "นั่ง" เพราะ V1 สับสนกับ "ยืน"`
  - ❌ `update model`

## 5. Participation & Collaboration (5 คะแนน)

| ระดับ | เกณฑ์ | หลักฐานที่ทีมควรมี |
|---|---|---|
| **4** | มีส่วนร่วมอย่างกระตือรือร้นและสม่ำเสมอตลอดกิจกรรม แสดงทักษะการทำงานเป็นทีม การสื่อสาร ความคิดริเริ่ม และการสนับสนุนสมาชิกในทีมได้อย่างยอดเยี่ยม | ทุกคนในทีมมี commit + Pull Request review กัน |
| **3** | มีส่วนร่วมในกิจกรรมอย่างสม่ำเสมอ และมีทักษะที่ดีในการทำงานเป็นทีม | ส่วนใหญ่ในทีมมี commit |
| **2** | มีส่วนร่วมในบางกิจกรรมและช่วยงานกลุ่มเมื่อได้รับการกระตุ้นหรือมอบหมาย | บางคนไม่มี commit |
| **1** | มีส่วนร่วมในกิจกรรมค่อนข้างน้อย | คนเดียว commit ทั้งหมด |

**Tips สำหรับนักเรียน:**
- ทุกคนในทีมต้องเป็น GitHub Contributor (มี commit อย่างน้อย 3 ครั้ง)
- ใช้บทบาท **3H** ชัดเจน:
  - **Hacker** → train/deploy/code
  - **Hipster** → data quality + UI/design
  - **Hustler** → documentation + present + idea

## 🎯 รวมคะแนน + การคำนวณ

```
คะแนนรวม = (Setup × 1.25) + (Core × 2.5) + (Data × 1.25) +
           (Debug × 1.25) + (Participation × 1.25)

ตัวอย่าง: ทีมได้ระดับ 4 ทุกหัวข้อ
= (4×1.25) + (4×2.5) + (4×1.25) + (4×1.25) + (4×1.25)
= 5 + 10 + 5 + 5 + 5
= 30 คะแนน ✓
```

## 🔍 เช็กตัวเองก่อนส่ง

- [ ] Commit history (จำนวน, ช่วงเวลา, ความชัดของ commit message)
- [ ] Contributors graph (ทุกคนมี commit ไหม)
- [ ] Pull requests / Issues (มี collaboration ไหม)
- [ ] README + docs ครบไหม
- [ ] Worksheets W1-W4 ครบไหม

- [ ] เวลาถาม "ทำไม X?" — ตอบได้ไหม

## ⚠️ Red Flags (ลดคะแนน)

- ❌ Copy code จาก ChatGPT/internet โดยไม่ credit
- ❌ Dataset เก็บใน 5 นาที (ไม่ diverse แน่นอน)
- ❌ Commit message แบบ "update update update"
- ❌ คนเดียวทำทั้งทีม
- ❌ Train accuracy 99% แต่ใช้จริงไม่ได้ (overfitting)

สรุปแบบสั้นที่สุด: ถ้าทีมอธิบายการตัดสินใจได้, มีหลักฐานใน repo, และ iterate จากผลทดสอบจริง คะแนนจะมาง่ายกว่าการไล่ accuracy อย่างเดียว
